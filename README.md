# xstocks-mvp

Buy tokenized stocks (xStocks) → deposit as collateral on Kamino's xStocks Market.

Stack:
- **Privy** — embedded Solana wallets, MoonPay onramp
- **Jupiter** — USDC → xStock swaps via aggregator
- **Kamino KTX REST API** — deposit transactions, no SDK needed
- **Next.js 14** — frontend + API routes (the only "backend" you have)

## Setup

```bash
npm install
cp .env.example .env.local
# Fill in PRIVY_APP_ID, PRIVY_APP_SECRET, SOLANA_RPC (Helius)

# Discover the Kamino xStocks market and reserve addresses
npm run bootstrap
# This writes config/reserves.json and prints NEXT_PUBLIC_XSTOCKS_MARKET_PUBKEY
# Paste that into .env.local

npm run dev
```

## What's wired up

1. **Sign in** (email/Google/Apple via Privy) → embedded Solana wallet auto-provisioned
2. **Add funds** → MoonPay (need to enable plugin in Privy console first)
3. **Buy** → `/api/jupiter/quote` → user reviews → `/api/jupiter/swap` → Privy signs → submit
4. **Lend** → `/api/kamino/deposit` (KTX) → Privy signs → submit. Opens an obligation, no borrow.
5. **Positions** → reads wallet (Token-2022!) + Kamino obligations API

## The footguns I hit so you don't have to

**xStocks are Token-2022, not classic SPL.** Every balance read uses `TOKEN_2022_PROGRAM_ID`. If you see "no balance" after a successful buy, that's why.

**Kamino market addresses are not stable across v2 rollout.** That's why `bootstrap-reserves.ts` queries the live API instead of hardcoding. Re-run it if reserves get added or rotated.

**KTX response field names.** The endpoint returns the unsigned tx but field naming has shifted between minor versions (`transaction` vs `tx` vs `serializedTransaction`). The deposit route normalizes all three. If a new variant ships, add it there.

**Jupiter slippage on thin off-hours liquidity.** Default is 50bps (0.5%). For weekend trades on lower-volume xStocks (anything outside top 5), bump to 100-200bps in the UI or trades will fail.

**MoonPay plugin must be enabled in the Privy console** — it requires a KYB review by Privy. Apply early.

## What's NOT in here (intentional cuts)

- Fee-payer / gas sponsorship — works without it for now since swaps grab SOL from the wrap process, but for a real launch add a Privy managed fee-payer wallet so users never need SOL
- Borrow flow — Story B for v2
- Hyperliquid — v2
- Real database — Supabase mirror is straightforward, didn't add it to the scaffold to keep surface area minimal
- Real-time prices — currently shows price from the Jupiter quote at request time. Add Chainlink/Pyth or just poll Jupiter quote every N seconds when you build the watch-list view

## Project structure

```
app/
  page.tsx                  → landing + login
  buy/page.tsx              → quote → swap flow
  positions/page.tsx        → wallet balances + lend action
  api/
    jupiter/quote/          → GET quote
    jupiter/swap/           → POST build swap tx
    kamino/deposit/         → POST build KTX deposit tx
    positions/              → GET wallet + obligation state
config/
  xstocks.ts                → top 10 supported tokens
  reserves.json             → generated by bootstrap, gitignored in real repo
lib/
  reserves.ts               → load reserves.json
scripts/
  bootstrap-reserves.ts     → one-time discovery from Kamino API
```
